# 트러블슈팅 (Troubleshooting)

Open-RMF 오피스 데모를 돌리면서 실제로 겪은 문제와 해결법.
환경: **Ubuntu 24.04 + ROS 2 Jazzy + Gazebo Harmonic(gz-sim 8)**.

> 셋업/실행 전체 흐름은 [README.md](./README.md) 참고.

---

## ① Gazebo가 열렸다가 바로 꺼짐 — `Unable to find uri[model://Open-RMF/TinyRobot]`

**증상**
- `ros2 launch rmf_demos_gz office.launch.xml` 실행 시 Gazebo 창이 잠깐 떴다가 바로 닫힘.
- 런처 로그 끝부분에 아래가 찍히고 gz가 종료됨:
  ```
  [gz] [Err] Error Code 14: ... Msg: Unable to find uri[model://Open-RMF/TinyRobot]
  [gz] [Err] Error Code 14: ... Msg: Unable to find uri[model://Open-RMF/TinyRobot]
  [INFO] [gz]: process has finished cleanly
  ```
- ⚠️ **그래픽 문제 아님.** RViz는 OpenGL 4.6으로 정상 로딩되고, gz도 Wayland 감지 후
  알아서 `QT_QPA_PLATFORM=xcb`를 잡음. 진짜 원인은 **로봇 모델을 못 찾아서**임.

**원인**
- 월드 파일(apt로 깐 `rmf_building_map_tools`가 빌드 때 생성)은 로봇을
  **`model://Open-RMF/TinyRobot`** (Fuel 스타일, `Open-RMF/` 접두사)로 참조함.
- 그런데 소스 `rmf_demos_assets`(jazzy)는 모델을 접두사 없이 **`models/TinyRobot`** 로 제공.
- gz는 `model://Open-RMF/TinyRobot`을 `GZ_SIM_RESOURCE_PATH`의 각 경로 아래에서
  `Open-RMF/TinyRobot/` 로 찾는데, 로컬엔 `Open-RMF/` 한 단계가 없어서 실패.
- 즉 **월드 생성기(apt) ↔ 모델 에셋(소스) 간 경로 규칙 불일치**.
- 참고: 오피스 월드의 다른 모델들(`Coke`, `OfficeChair`, `MiR100` 등)은 접두사 없는
  `model://...` 라서 정상적으로 찾힘. 문제 되는 건 `Open-RMF/` 접두사가 붙은 로봇뿐.

**해결**
- 오피스 런처는 `GZ_SIM_RESOURCE_PATH`에 `~/.gazebo/models`를 자동으로 추가함.
- 그래서 거기에 `Open-RMF/` 네임스페이스 폴더를 만들고, 로컬 로봇 모델을 심볼릭 링크하면 됨.
  ```bash
  mkdir -p ~/.gazebo/models/Open-RMF
  for m in ~/open-rmf-test/rmf_ws/src/rmf_demos/rmf_demos_assets/models/*; do
    ln -sfn "$m" ~/.gazebo/models/Open-RMF/"$(basename "$m")"
  done
  ```
- **검증**:
  ```bash
  ls ~/.gazebo/models/Open-RMF/TinyRobot/model.config   # 파일이 나오면 OK
  ```
- 이러면 `model://Open-RMF/TinyRobot` → `~/.gazebo/models/Open-RMF/TinyRobot` 로 풀려서
  로봇이 정상 스폰되고 Gazebo 창이 유지됨.
- (모델 전부를 링크해뒀으므로 hotel/airport 등 다른 데모의 `Open-RMF/<로봇>`도 같이 해결됨.)

## ② `Connection to ws://localhost:7878 failed` 에러가 반복됨

**증상**
```
[rmf_task_dispatcher] Attempting to connect to ws://localhost:7878
[rmf_task_dispatcher] Connection to ws://localhost:7878 failed. ... Reconnecting in 1s
[fleet_adapter] Connection not yet established
```

**원인**
- 실행 시 `server_uri:=ws://localhost:7878` 인자를 붙이면 **rmf-web 대시보드**의
  웹소켓 서버에 접속을 시도함. 대시보드를 안 띄웠으니 접속이 계속 실패.

**해결**
- 대시보드를 안 쓸 거면 **`server_uri` 인자를 빼고** 실행:
  ```bash
  ros2 launch rmf_demos_gz office.launch.xml
  ```
- 데모 자체 동작(로봇 이동, 태스크 처리)엔 **전혀 지장 없는 무해한 에러**임.
- 대시보드까지 쓰려면 별도로 [`rmf-web`](https://github.com/open-rmf/rmf-web)을 띄워야 함.

## ③ `package 'rmf_demos_gz_classic' not found`

**증상**
```
Package 'rmf_demos_gz_classic' not found
```

**원인**
- Jazzy에서는 **Gazebo Classic이 단종(EOL)** → `rmf_demos_gz_classic`을 빌드하지 않음.

**해결**
- 새 Gazebo(Harmonic)용 패키지인 **`rmf_demos_gz`** 를 사용:
  ```bash
  ros2 launch rmf_demos_gz office.launch.xml
  ```
- 옛날 튜토리얼/블로그의 `_classic` 명령어를 그대로 복사하지 말 것.

## ④ (참고) Wayland 세션에서의 GUI

- gz Harmonic은 Wayland를 감지하면 **자동으로 `QT_QPA_PLATFORM=xcb`** 를 설정함
  (로그: `Detected Wayland. Setting Qt to use the xcb plugin`).
- 따라서 Wayland 세션이어도 Gazebo GUI는 정상 동작함.
- "창이 열렸다 꺼지는" 증상의 원인은 Wayland가 아니라 **①번(로봇 모델 못 찾음)** 이었음.
